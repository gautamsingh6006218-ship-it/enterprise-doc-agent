# Enterprise Document AI Agent

Production-grade RAG (Retrieval-Augmented Generation) pipeline. Ingest documents from files, Confluence, SharePoint, Jira, or a watched directory. Ask questions in natural language ‚Äî answers are generated by a local Llama model using retrieved document context.

---

## What It Does

```
Upload PDF / DOCX / TXT / HTML / PPTX / XLSX / ...
         ‚îÇ
         ‚ñº
[1] Ingest       ‚Üí extract text (PyMuPDF ‚Üí Docling ‚Üí OCR fallback)
[2] Preprocess   ‚Üí clean, normalize, detect language & metadata
[3] Chunk        ‚Üí 512-token chunks with sentence-window links
[4] Embed        ‚Üí BGE-M3 dense + sparse vectors ‚Üí pgvector
         ‚îÇ
         ‚ñº
Ask a question
         ‚îÇ
         ‚ñº
[5] Encode query ‚Üí BGE-M3 1024-dim vector
[6] Retrieve     ‚Üí dense + BM25 keyword search ‚Üí RRF fusion ‚Üí reranker
[7] Answer       ‚Üí Llama 3.2 (via Ollama, runs locally)
```

## Setup ‚Äî Step by Step

### Step 1 ‚Äî Clone and create virtual environment

```bash
git clone <repo-url>
cd Enterprise-doc-agent

python3 -m venv .venv
source .venv/bin/activate          # macOS / Linux
# .venv\Scripts\activate           # Windows
```

> **Note (macOS with Anaconda):** If `which python` still shows Anaconda after activating,  
> use `.venv/bin/python` directly instead of `python` for all commands below.

---

### Step 2 ‚Äî Install Python dependencies

```bash
pip install -r requirements.txt

# transformers must be pinned below 5.0 (FlagEmbedding reranker compatibility)
pip install "transformers>=4.47,<5.0"
```

---

### Step 3 ‚Äî Start PostgreSQL + pgvector (Docker)

```bash
docker compose up -d
```

This starts a PostgreSQL 16 container with the pgvector extension and automatically creates the `document_chunks` and `documents` tables.

Verify it is healthy:

```bash
docker compose ps
# enterprise_doc_db should show "healthy"
```

---

### Step 4 ‚Äî Pull the Llama model (Ollama)

Make sure Ollama is running, then pull the model:

```bash
ollama pull llama3.2
```

This downloads ~2 GB. Run once, reused on every startup.

---

## Using the App

### Chat (ask questions)

1. Go to the **Chat** tab
2. Type any question about your documents
3. The answer is generated by Llama 3.2 using the most relevant document chunks
4. Click **"N sources"** below an answer to see exactly which document chunks were used

### Documents (upload & manage)

1. Go to the **Documents** tab
2. Drag and drop files onto the upload zone ‚Äî or click **Browse**
3. Supported formats: PDF, DOCX, PPTX, XLSX, CSV, TXT, MD, HTML, EML, PNG, JPG, JPEG, TIFF
4. Click **Upload** ‚Äî each file is processed through the full pipeline
5. Status badges:
   - `completed` ‚Äî indexed and searchable
   - `duplicate` ‚Äî identical file already ingested, skipped
   - `failed_*` ‚Äî pipeline error (hover for details)
6. Click the trash icon to delete a document (also removes its vectors from pgvector)

### Re-uploading a changed document

If you upload a file with the **same name** but **different content**, the system automatically deletes the old version and indexes the new one. No manual deletion needed.

## API Reference

All endpoints require `Authorization: Bearer <token>`.

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/ingest` | Upload a single file |
| `POST` | `/ingest/batch` | Upload multiple files at once |
| `POST` | `/query` | Semantic search + optional LLM answer |
| `GET` | `/documents` | List all documents for your tenant |
| `GET` | `/documents/{id}` | Get pipeline status of one document |
| `DELETE` | `/documents/{id}` | Delete document + all its chunks |
| `POST` | `/sync/{source}` | Trigger connector sync (confluence/jira/sharepoint/wiki) |
| `POST` | `/auth/token` | Generate a dev JWT token |
| `GET` | `/health` | Health check (no auth required) |

**Query with LLM answer:**

```bash
curl -X POST http://localhost:8000/query \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"query": "what is the leave policy?", "generate_answer": true}'
```

**Upload a file:**

```bash
curl -X POST http://localhost:8000/ingest \
  -H "Authorization: Bearer <token>" \
  -F "file=@/path/to/document.pdf" \
  -F "visibility=public"
```

## Auto-Sync Features

### Watch a local folder

Set `WATCH_DIR` in `.env` to a folder path. Any file dropped into that folder is automatically ingested. Changed files are automatically re-ingested (old version replaced).

```env
WATCH_DIR=/Users/you/Documents/company-docs
WATCH_TENANT_ID=acme
WATCH_RECURSIVE=true
```

### Scheduled connector sync

Set `SYNC_SOURCES` and `SYNC_INTERVAL_MINUTES` in `.env`. The server polls each connector on a background schedule and ingests new/updated documents.

```env
SYNC_SOURCES=confluence,jira
SYNC_INTERVAL_MINUTES=60
SYNC_TENANT_ID=acme
```

## Architecture Decisions

| Decision | Reason |
|---|---|
| PostgreSQL + pgvector (not Qdrant/Pinecone) | Vectors and RBAC metadata in one DB ‚Äî no dual-write problem |
| BGE-M3 (not separate dense + BM25 models) | One model produces both ‚Äî no second service to manage |
| bge-reranker-v2-m3 local (not Cohere Rerank API) | Enterprise documents must not leave the organisation |
| Llama 3.2 via Ollama (not OpenAI/Anthropic) | Fully local ‚Äî no data sent to external APIs |
| Result pattern everywhere (never raises) | All services return `XxxResult(success, data, error)` ‚Äî safe in async workers |
| RBAC enforced in SQL (not post-filter) | Guarantees exactly top_k RBAC-valid results, not top_k minus filtered |
